### P0-1: `export.go` — Multiple Type Mismatches in Audit Log CSV Export (Committed)

**Files & Lines**:

- `apps/backend/internal/service/export.go:119-125` (the loop)
- `apps/backend/internal/domain/admin.go:246-263` (domain model)
- `apps/backend/internal/repository/admin_audit_repo.go:92-93` (how data is actually produced)

**Compiler Errors** (verbatim from `go build` and `go vet`):

```
internal/service/export.go:122:21: cannot convert l.Changes (variable of type []domain.ChangeEntry) to type string
internal/service/export.go:124:24: cannot use l.ID (variable of type int64) as string value in array or slice literal
internal/service/export.go:124:55: cannot use l.Action (variable of string type domain.AuditAction) as string value in array or slice literal
internal/service/export.go:124:113: cannot use l.Severity (variable of string type domain.AuditSeverity) as string value in array or slice literal
```

**Root Cause Analysis**:

The function `exportAuditLogs` was written against a mental model of the `AuditLog` struct that never existed in this codebase.

Actual schema (committed at `HEAD`):

```go
type AuditLog struct {
    ID         int64         `json:"id"`
    ActorID    string        `json:"actorId"`
    ActorEmail string        `json:"actorEmail"`
    Action     AuditAction   `json:"action"`     // defined type: type AuditAction string
    TargetType string        `json:"targetType"`
    TargetID   string        `json:"targetId"`
    Changes    []ChangeEntry `json:"changes,omitempty"`  // slice of structs
    IPAddress  string        `json:"ipAddress,omitempty"`
    Severity   AuditSeverity `json:"severity"`   // defined type
    CreatedAt  time.Time     `json:"createdAt"`
}
```

What the export code assumed:

- `ID`, `Action`, `Severity` were all `string`
- `Changes` was `[]byte` or `string` (hence the `string(l.Changes)` attempt)

The code even has a defensive `if l.Changes != nil` check, which is meaningless for a slice (slices are never nil after JSON unmarshal in this context, and the conversion would still be illegal).

**Evidence Chain**:

1. Domain definition proves the real types (`admin.go:252`)
2. Repository correctly scans using pgx into these exact types (`admin_audit_repo.go:92`)
3. Export function ignores the repository output types completely

**Git History**:

- Introduced in `680016c` ("implement ... exports")
- Survived `87e1475`, `2780c1a`, `b6c5755` — all claiming to have resolved "35+", "43", and "120" issues respectively

**Impact**: The entire audit export feature is dead on arrival. Any attempt to call the export endpoint will fail at compile time for the whole backend.

## P1 — Logic & Correctness Bugs

---

### P1-1: Daily Budget Reset Uses Wrong Date Comparison (Duplicate Implementations)

**Locations** (two independent implementations of the identical bug):

1. `internal/service/credits.go:197-205` — inside the `LogAndDeduct` transaction
2. `internal/service/credits.go:126-134` — inside the `checkBudget` helper (called by `CheckBalance` and `LogAndDeduct`)

**Buggy logic** (both places):

```go
if now.Day() != reset.Day() || now.Month() != reset.Month() || now.Year() != reset.Year() {
    newDaily = cost   // or c.DailySpent = 0
}
```

**Root Cause**:
The condition assumes that crossing a day boundary always produces a different `.Day()` value. This is false across month boundaries.

**Concrete Counter-Example**:

- Last reset recorded: 2026-01-15
- Today: 2026-02-15
- `now.Day() == 15`, `reset.Day() == 15`
- Condition is false → daily spend is **not** reset
- User carries January 15th's daily spend into February

The monthly reset logic right next to it (`now.Month() != reset.Month()`) is correct. The daily logic is not.

**Impact**:

- Users can exceed (or be incorrectly blocked by) daily budgets when crossing months on dates that share the same day number.
- Financial correctness of the entire credit/budget system is compromised on ~28–31 days per year per user who has daily budgets configured.

**Why This Is Especially Bad**:
This exact class of date bug has appeared in production billing systems for decades. It is a well-known trap. The fact that it exists in **two places** after multiple claimed "comprehensive audits" indicates insufficient scrutiny of the credits domain.

### P1-2: Middleware Body Consumption Ordering Risk (QuotaCheck + TransformMiddleware)

**Files**:

- `internal/middleware/quota.go:247-252`
- `internal/middleware/transform.go:32-46`

Both middlewares unconditionally read `r.Body` with `io.ReadAll`, close the original body, and replace it with a `bytes.Reader` copy.

**Current registration order** (from `routes.go` and `cmd/api/main.go` wiring):

- `QuotaCheck` (or `RedisQuota`) is registered early in the auth+quota chain
- `TransformMiddleware` is registered globally very early (before most other middleware)

**Risk**:
If `TransformMiddleware` runs **after** `QuotaCheck` on chat completion requests:

- Quota middleware reads the body to extract the model for token estimation
- It restores a copy
- Transform then reads the (possibly already-modified) body again
- The model name seen by quota estimation may differ from the final model after transformation (system prompt injection, model aliasing, etc.)

Even if current registration order happens to be safe today, the two middlewares are **mutually destructive** if their relative order ever changes.

**Evidence**:

- QuotaCheck does `io.ReadAll(r.Body)`, `r.Body.Close()`, two `NopCloser(bytes.NewReader(body))` operations
- TransformMiddleware does the exact same pattern

This is a classic "multiple body readers" footgun that has caused subtle production bugs in many Go APIs.

### P1-3: Token Blacklist Treats Raw API Keys as Revocable Tokens

**File**: `internal/middleware/token_blacklist.go:42-44`

```go
if k := GetAPIKey(r); k != nil {
    tokenStr = k.Key   // raw API key value used as blacklist identifier
}
```

**Concern**:
The blacklist system was primarily designed for JWT revocation (see migration `018_token_blacklist.sql` and the `TokenBlacklistService`).

Using the raw, long-lived API key string as the blacklist "token" means:

- If an API key is compromised and then rotated by the user, the old key value may still be in the blacklist table
- The blacklist table now mixes two completely different identifier spaces (JWTs and API keys)
- There is no clear revocation semantic for "revoke this specific API key" vs "this JWT is no longer valid"

**Not necessarily a security hole today**, but a design smell with future blast radius.

## P2 — Architectural Debt & Quality Issues

---

### P2-1: Inconsistent and Incomplete Dependency Injection (Confirmed & Expanded)

Already documented in the first version. Additional evidence from this pass:

- `handler.New()` creates `exportSvc` with three fresh, uncached repositories (line 84)
- The same pattern exists for `fineTuningSvc`, `providerPluginSvc`, `rbacSvc`, `budgetSvc`, `comparisonSvc`, etc.
- Only `userRepo`, `keyRepo`, and `creditsRepo` ever receive the `repoCache` (services.go:45-47)

This means the majority of admin, audit, batch, comparison, fine-tuning, and export operations bypass the caching layer entirely.

---

### P2-2: Duplicate Budget Reset Logic (Technical Debt Amplification)

The identical faulty date logic exists in two methods inside the same file (`checkBudget` and the transaction inside `LogAndDeduct`).

Any future fix must be applied in two places, or the bug will regress.

This is a direct violation of DRY and increases the chance that the bug will be only partially fixed.

---

### P2-3: Handler Constructor Creates Too Many Services Internally

`handler.New()` is 20 lines long and creates 15+ services using raw repository constructors. This makes the Handler a god object during construction and makes unit testing of most admin/export/batch/fine-tuning flows structurally difficult without a real database.

---

### P2-4: Batch Service Fire-and-Forget Goroutine With No Error Channel or Persistence of Failures

In `batch.go:48`:

```go
go s.process(job.ID, items)
```

The async processing goroutine is launched with no way for the caller (or monitoring) to observe whether it ever completed, panicked, or got stuck. Failures are only logged inside `process` (presumably). If the process goroutine dies silently, the job remains in `pending` or `processing` state forever.

This is a common source of "zombie jobs" in batch systems.

## Conclusion of Deep Re-Audit

The second pass confirmed that the initial findings were not isolated incidents. The same class of error (incomplete implementation, missing updates to callers/tests, incorrect date arithmetic in financial logic) appears repeatedly across different domains (exports, Stripe, credits/budgets, batch, middleware composition).

The fact that these survived multiple prior "exhaustive" audits suggests that previous reviews may have focused more on security checklists and high-level architecture than on exhaustive compilation + logic verification of every new feature path.

**Recommendation**: Before any further feature work, the build must be made green and the budget reset logic must be centralized + correctly implemented with proper tests.
